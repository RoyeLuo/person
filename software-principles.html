S - 单一职责原则 (SRP)

Single Responsibility Principle

核心思想：一个类只负责一项职责，只有一个引起它变化的原因。
好处：类的复杂度降低、可读性提高、可维护性提高、变更引起的风险降低。

违反原则

UserService {
    saveUser() { // 保存用户 }
    sendEmail() { // 发送邮件 }
    generateReport() { // 生成报表 }
    logActivity() { // 记录日志 }
}

遵循原则

UserService { saveUser() {} }
EmailService { sendEmail() {} }
ReportService { generateReport() {} }
LoggerService { logActivity() {} }
O - 开闭原则 (OCP)

Open Closed Principle

核心思想：对扩展开放，对修改关闭。用抽象构建框架，用实现扩展细节。
好处：提高代码的可复用性和可维护性，降低变更带来的风险。

违反原则

calculateFee(type) {
    if (type === 'student') { // 学生票 }
    if (type === 'adult') { // 成人票 }
    // 每新增类型都要修改这里
}

遵循原则

interface Ticket {
    getFee(): number;
}
StudentTicket implements Ticket { ... }
AdultTicket implements Ticket { ... }
// 新增类型只需扩展，无需修改
L - 里氏替换原则 (LSP)

Liskov Substitution Principle

核心思想：子类可以扩展父类的功能，但不能改变原有父类函数的行为。
核心理解：子类对象能够替换父类对象，而程序行为不变。

违反原则

Rectangle {
    setWidth(w) { this.width = w; }
    setHeight(h) { this.height = h; }
}
Square extends Rectangle {
    // 正方形宽高必须相等，行为与父类不一致！
    setWidth(w) { this.width = w; this.height = w; }
}

遵循原则

interface Shape {
    area(): number;
}
Rectangle implements Shape { ... }
Square implements Shape { ... }
// 各自独立，互不影响
I - 接口隔离原则 (ISP)

Interface Segregation Principle

核心思想：建立单一接口，尽量细化接口，类之间的依赖关系建立在最小接口上。
核心理解：客户端不应依赖它不需要的接口，一个类对另一个类的依赖应建立在最小的接口上。

违反原则

interface IAnimal {
    fly();
    run();
    swim();
}
Dog implements IAnimal {
    fly() { throw "狗不会飞"; }
}

遵循原则

interface IFly { fly(); }
interface IRun { run(); }
interface ISwim { swim(); }
Dog implements IRun, ISwim { ... }
D - 依赖倒置原则 (DIP)

Dependency Inversion Principle

核心思想：高层模块不应该依赖低层模块，两者都应该依赖抽象。抽象不应该依赖细节，细节应该依赖抽象。
核心理解：面向接口编程，而不是面向实现。

违反原则

OrderService { private MySQLRepository repo; save() { repo.save(); } } // 直接依赖具体实现，耦合度高

遵循原则

OrderService { private IRepository repo; save() { repo.save(); } } MySQLRepository implements IRepository { ... } // 依赖抽象，便于扩展和替换
C - 合成复用原则 (CRP)

Composite Reuse Principle

核心思想：尽量使用对象组合(has-a)/聚合，而不是继承(is-a)来达到复用目的。
核心理解：继承会让类之间的关系变得复杂，组合更灵活。

违反原则

ClassA { methodA() {} } ClassB extends ClassA { // 继承关系过于紧密 } // ClassB 的改变会影响 ClassA

遵循原则

ClassA { methodA() {} } ClassB { private ClassA obj; methodB() { obj.methodA(); } } // 组合关系，更灵活，可动态替换
