缓存基础

Redis（Remote Dictionary Server）是一个开源的内存数据结构存储系统，常用作数据库、缓存和消息队列。

数据类型

String

最基本的键值类型，最大512MB

计数器Session

Hash

字段值对，适合存储对象

用户信息商品详情

List

有序字符串列表，LPUSH/RPOP

消息队列最新列表

Set

无序唯一成员集合

标签去重

ZSet

有序集合，带分数排序

排行榜权重队列

Bitmap

位图，适合大规模状态

签到活跃用户

常用命令

# 字符串操作
SET user:1001:name "张三" # 设置值
GET user:1001:name           # 获取值
SETNX lock:order:123 "1"  # 分布式锁（需配合EXPIRE）
INCR page:view:20240101     # 原子递增

# Hash操作
HSET product:1001 name "手机" price 2999
HGET product:1001 name
HGETALL product:1001

# 有序集合 - 排行榜
ZADD leaderboard 1500 "用户A"
ZINCRBY leaderboard 100 "用户A"  # 增加分数
ZREVRANGE leaderboard 0 9 WITHSCORES  # 获取前10

# 过期时间
EXPIRE session:abc 3600  # 1小时后过期
TTL session:abc              # 查看剩余时间

缓存策略

缓存模式

模式原理适用场景一致性
Cache Aside读：先缓存后DB
写：先DB后删缓存		读多写少最终一致
Read Through应用只读缓存，缓存负责加载简单读场景强一致
Write Through写操作同时写缓存和DB强一致性需求强一致
Write Behind异步批量写DB高并发写最终一致

缓存问题与解决方案

缓存穿透：大量请求查询不存在的数据
解决：布隆过滤器 / 缓存空值 / 限流保护
缓存击穿：热点key过期瞬间大量请求击穿到DB
解决：互斥锁 / 永不过期+异步更新 / 热点数据永不过期
缓存雪崩：大量缓存同时过期或Redis宕机
解决：过期时间加随机值 / 多级缓存 / Redis高可用 / 限流降级

过期策略

volatile-lru 仅删除过期键中LRU最近最少使用

allkeys-lru 所有键中LRU最近最少使用

volatile-ttl 删除即将过期的键

noeviction 不删除，返回错误（默认）

分布式锁

基于Redis实现分布式锁，是分布式系统中常用的同步机制。

# 1. 基础加锁（NX保证原子性，PX设置过期时间）
SET lock:order:123 "unique_value" NX PX 30000

# 2. 释放锁（Lua脚本保证只能释放自己的锁）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
Redisson 客户端实现： 
// Java示例
RLock lock = redisson.getLock("order:123");
lock.lock(30, TimeUnit.SECONDS);
try {
    // 业务逻辑
} finally {
    lock.unlock();
}

分布式锁要点

  • 互斥性：同一时刻只有一个客户端能获取锁
  • 死锁避免：设置锁的自动过期时间
  • 安全性：只能释放自己的锁（value唯一标识）
  • 容错性：RedLock算法（多Redis实例多数派获取锁）

消息队列

Redis的List和Stream类型可以用作轻量级消息队列。

# 生产者 - LPUSH消息
LPUSH queue:orders "{\"orderId\":123,\"amount\":99}"

# 消费者 - BRPOP阻塞读取（超时0表示无限等待）
BRPOP queue:orders 0

# 消息ID - 每次自动生成递增ID
XADD stream:orders * orderId 123 amount 99

# 消费组 - 消息确认机制
XGROUP CREATE stream:orders mygroup $ # 创建消费组，从最新消息开始
XREADGROUP GROUP mygroup consumer1 COUNT 10 STREAMS stream:orders >
XACK stream:orders mygroup 消息ID
List vs Stream 对比：
特性ListStream
消息ID需手动生成自动生成唯一ID
消费组不支持支持（N个消费者负载均衡）
消息确认需手动处理XACK确认
广播模式不支持支持（多个消费组独立消费）
适用场景简单队列可靠消息、发布订阅

Redis集群

主从复制

一主多从架构，实现数据冗余和读写分离。

# 从节点配置
replicaof 192.168.1.100 6379
replica-read-only yes  # 从节点只读

# 全量同步（RDB + AOF）
# 增量同步（命令传播）
# 无磁盘复制（repl-diskless-sync yes）

Sentinel哨兵

监控主从节点，自动故障转移，对客户端透明。

  • 监控：心跳检测主节点存活
  • 通知：故障时通知应用
  • 自动故障转移：选举新主节点
  • 配置提供者：返回当前主节点地址

Cluster集群

数据分片（16384个槽），每节点存储部分数据。

# 集群节点数：至少6个（3主3从）
# 槽分配：每个主节点约5461个槽
# 故障转移：主节点宕机后从节点自动升级

# 集群命令
CLUSTER NODES       # 查看集群节点
CLUSTER INFO        # 集群状态
CLUSTER SLOTS       # 槽分配情况

Redis vs Memcached

特性RedisMemcached
数据类型丰富（String/Hash/List/Set/ZSet等）仅String
持久化支持（RDB + AOF）不支持
集群Cluster原生支持需第三方（membase）
内存管理支持虚拟内存固定内存分配
性能稍低（数据结构更丰富）纯KV更高

最佳实践

  • Key命名规范：业务:模块:标识 如 user:profile:1001
  • 避免大Key：单个Value不超过10MB，Hash/Set建议不超过10000字段
  • 合理过期时间：热点数据短、配置数据长、Log类 TTL
  • Pipeline批量：减少网络往返，提升性能
  • 连接池复用：使用连接池而非频繁创建连接
  • 监控指标：内存使用、连接数、命中率、慢查询
  • 安全配置：开启密码认证、禁用危险命令（FLUSHDB/FLUSHALL/KEYS）
Pipeline vs Transaction (MULTI/EXEC)：
Pipeline：批量执行命令，原子性无保证，返回顺序固定
Transaction：保证原子性，但无回滚能力（Watch乐观锁除外）
