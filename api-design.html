RESTful API 基础

REST（Representational State Transfer）是一种架构风格，核心是资源导向的URL设计。

HTTP方法对应操作

方法语义幂等安全示例
GET查询资源✓✓GET /users/1001
POST创建资源✗✗POST /users
PUT完整更新✓✗PUT /users/1001
PATCH部分更新✗✗PATCH /users/1001
DELETE删除资源✓✗DELETE /users/1001
幂等性：同一请求执行一次和执行多次效果相同
安全性：请求是否改变服务器状态

URL命名规范

# 正确示例
GET    /users              # 用户列表（复数名词）
GET    /users/1001         # 用户详情（资源+ID）
GET    /users/1001/orders  # 用户的订单（嵌套资源）
POST   /users              # 创建用户
PUT    /users/1001         # 更新用户
DELETE /users/1001         # 删除用户

# 错误示例 ❌
GET /getUsers             # 避免动词
GET /userInfo?id=1001     # 资源名应为名词
POST /createUser          # 避免动词
DELETE /deleteUser/1001   # 避免动词

请求与响应

请求参数设计

# Query参数 - 过滤、分页、排序
GET /users?status=active&page=1&size=20&sort=createdAt,desc
GET /users?city=beijing&role=admin
GET /products?minPrice=100&maxPrice=500

# Path参数 - 资源标识
GET /users/1001
GET /orders/20240115/123

# Header参数 - 认证、分页、Content-Type
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
Content-Type: application/json
Accept-Language: zh-CN
X-Request-ID: uuid-xxx

# Body参数 - POST/PUT/PATCH请求体
{
    "username": "zhangsan",
    "email": "zhang@example.com",
    "password": "********",
    "profile": {
        "age": 28,
        "city": "Beijing"
    }
}

统一响应格式

# 成功响应

        "name": "张三",

# 分页响应
{

    "data": {
        "list": [...],
        "pagination": {
            "page": 1,
            "size": 20,
            "total": 1000,

# 错误响应

    "code": 40001,
    "message": "参数校验失败",

HTTP状态码

2xx 成功

200 OK 201 Created 204 No Content

4xx 客户端错误

400 Bad Request 401 Unauthorized 403 Forbidden 404 Not Found 422 Unprocessable

5xx 服务端错误

500 Internal Error 502 Bad Gateway 503 Unavailable 504 Timeout

版本控制

API需要演进但要保持向后兼容，版本控制是重要的策略。

# 方式1：URL路径（最常用）
GET /api/v1/users
GET /api/v2/users

# 方式2：Query参数
GET /api/users?version=2

# 方式3：Header（推荐用于大版本）
GET /api/users
Accept: application/vnd.myapp.v2+json

# 方式4：子域名
GET https://v2.api.example.com/users
建议：小版本用向后兼容的升级，大版本才breaking change

认证与授权

JWT认证流程

# 1. 登录获取Token
POST /auth/login
Request: { "username": "xxx", "password": "xxx" }
Response: { "accessToken": "xxx", "refreshToken": "xxx", "expiresIn": 7200 }

# 2. 请求时携带Token
GET /api/users
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

# 3. Token结构（Payload示例）
{
    "sub": "1001",           # 用户ID
    "username": "zhangsan",
    "roles": ["admin", "user"],
    "iat": 1705123456,     # 签发时间
    "exp": 1705127056      # 过期时间
}

GraphQL vs REST

特性RESTGraphQL
数据获取固定端点返回固定结构客户端指定需要字段
请求次数多个资源需多次请求一次请求获取多资源
版本控制通常需要v1/v2字段演进，无需版本
缓存HTTP缓存天然支持需自行实现缓存
学习成本简单直观概念较多
适用场景简单CRUD、移动API复杂聚合、移动端、数据平台
# GraphQL 示例
# 查询
query {
    user(id: "1001") {
        name
        email
        orders(limit: 5) {
            id
            amount
            createdAt
        }
    }
}

# 变更
mutation {
    createUser(input: {
        name: "张三"
        email: "zhang@example.com"
    }) {
        id
        name
    }
}

API安全

  • HTTPS：所有API必须使用HTTPS，防止中间人攻击
  • 参数校验：服务端必须校验所有输入参数，白名单过滤
  • 限流：防止DDoS和资源滥用（令牌桶/滑动窗口）
  • CORS：正确配置跨域策略
  • 敏感数据：日志脱敏，密码不在响应中返回
  • SQL注入：使用参数化查询
  • XSS：输出编码，Content-Type设置
  • CSRF：Token验证，非简单请求

限流响应头

X-RateLimit-Limit: 1000        # 请求总数限制
X-RateLimit-Remaining: 999    # 剩余请求数
X-RateLimit-Reset: 1705123800 # 重置时间戳
Retry-After: 3600           # 限流后需等待秒数
