MySQL优化

SQL语句优化

Query Optimization

优化建议

  • 避免 SELECT *，只查需要的字段
  • 用 EXPLAIN 分析执行计划
  • 批量插入代替循环单条
  • JOIN 控制在3张表以内
  • 避免 OFFSET 过大

应避免

  • WHERE 列上使用函数
  • LIKE '%xxx%' 全表扫描
  • OR 连接多个条件
  • IN 子句包含大量值
  • 循环中执行SQL
-- EXPLAIN 分析查询
EXPLAIN SELECT u.name, o.amount FROM users u INNER JOIN orders o ON u.id = o.user_id;

-- 优化分页：避免大 OFFSET
SELECT * FROM orders WHERE id > 100000 ORDER BY id LIMIT 20;

-- 批量插入
INSERT INTO users (name, email) VALUES
('张三', 'a@example.com'),
('李四', 'b@example.com');

索引优化

Index Optimization 
-- 单列索引
CREATE INDEX idx_user_status ON users(status);

-- 复合索引 (最左前缀原则)
CREATE INDEX idx_order_user_time ON orders(user_id, status, create_time);

-- 唯一索引
CREATE UNIQUE INDEX idx_user_email ON users(email);

-- 查看索引
SHOW INDEX FROM orders;

-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;
索引原则：区分度高的列优先；复合索引遵循最左前缀；避免在索引列上使用函数

表结构设计优化

Schema Optimization

数据类型选择

  • INT 代替 VARCHAR 存ID
  • TINYINT 存状态
  • DECIMAL 代替 FLOAT 存金额
  • 主键用 BIGINT 自增
  • 固定长用 CHAR，变长用 VARCHAR

分表策略

  • 按时间分表：日志、订单
  • 按用户ID哈希分表
  • 冷热数据分离
  • 单表控制在500万以内
-- 分区表示例：按月分区
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    amount DECIMAL(10,2),
    create_time DATETIME,
    INDEX idx_user_id (user_id)
) PARTITION BY RANGE (YEAR(create_time) * 100 + MONTH(create_time)) (
    PARTITION p202501 VALUES LESS THAN (202502),
    PARTITION p202502 VALUES LESS THAN (202503),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);

主从复制配置

Master-Slave Replication
原理：主库处理写操作，通过 binlog 传递给从库，从库重放 binlog 实现同步。
-- 主库 my.cnf
[mysqld]
server-id = 1
log_bin = /var/log/mysql/mysql-bin
binlog_format = ROW
sync_binlog = 1

-- 创建复制账号
CREATE USER 'repl'@'%' IDENTIFIED BY 'password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

-- 从库配置
[mysqld]
server-id = 2
relay_log = /var/log/mysql/mysql-relay-bin
read_only = 1

-- 连接主库
CHANGE MASTER TO MASTER_HOST='192.168.1.100', MASTER_USER='repl', 
    MASTER_PASSWORD='password', MASTER_LOG_FILE='mysql-bin.000001', MASTER_LOG_POS=123;
START SLAVE;
SHOW SLAVE STATUS\G

读写分离

Read-Write Splitting
思路：写走主库，读走从库，减轻压力，提高并发。
-- ProxySQL 中间件配置
INSERT INTO mysql_servers (hostgroup_id, hostname, port) VALUES
(0, '192.168.1.100', 3306),  -- 主库 写
(1, '192.168.1.101', 3306),  -- 从库1 读
(1, '192.168.1.102', 3306);  -- 从库2 读

-- 路由规则：SELECT走读组，其他走写组
INSERT INTO mysql_query_rules (rule_id, active, match_pattern, destination_hostgroup) VALUES
(1, 1, '^SELECT', 1);

-- C# 应用层路由示例
public MySqlConnection GetReadConnection() {
    return new MySqlConnection(_slaveConns[new Random().Next(_slaveConns.Length)]);
}
注意：主从同步有延迟；SELECT FOR UPDATE 必须走主库；事务内操作必须在同一库

MySQL集群方案

Clustering Solutions

Group Replication

官方原生多主集群，基于Paxos协议，自动故障切换。

SET GLOBAL group_replication_group_name = 'uuid';
SET GLOBAL group_replication_start_on_boot = ON;
START GROUP_REPLICATION;

InnoDB Cluster

官方推荐完整方案：Shell + Group Replication + Router。

shell.connect('admin@host');
var cluster = dba.createCluster('myCluster');
mysqlrouter --bootstrap admin@host

ProxySQL + Keepalived

经典高可用架构，VIP自动切换。

# VIP: 192.168.1.200
vrrp_instance VI_1 {
    state MASTER
    priority 100
    virtual_ipaddress { 192.168.1.200 }
}

ShardingSphere

分片集群方案，支持水平分片和读写分离。

# config-sharding.yaml
rules:
  - !SHARDING
    tables:
      t_order:
        actualDataNodes: ds_${0..1}.t_order
